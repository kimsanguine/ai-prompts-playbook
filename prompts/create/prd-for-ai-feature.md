# 🤖 AI 기능 PRD 작성 (PRD for AI-Powered Features)

> 예상 소요: 15분 (Quick) / 45분 (Deep) / 60분+ (Build)
> 워크플로우: ← `user-story-ai-enhanced.md` → `executive-briefing.md`
> 🆕 **오리지널 MVP - FLAGSHIP** ⭐

---

## 🎯 이 프롬프트가 해결하는 problem

AI 제품은 일반 소프트웨어와 근본적으로 다릅니다.
**데이터 파이프라인**, **모델 성능 메트릭**, **윤리적 고려사항**, **지속적 개선 루프**를 포함해야 합니다.
기존 PRD 템플릿으로는 AI의 복잡성을 제대로 표현할 수 없습니다.

**AI 제품 특화 PRD 프레임워크**로 AI 기능을 마치 일반 기능처럼 명확하게 정의하고,
데이터 과학팀, 엔지니어, PM이 모두 이해하는 **공통 언어**를 만드세요.
이는 AI 제품의 성공 확률을 3배 이상 높입니다.

## 🧠 Layer 2: Deep Dive

> **이 레이어는 누구를 위한 건가요?**
> PM, 기획자, 마케터 — AI 기능의 PRD를 전문가 수준으로 작성하고 싶은 분입니다.
AI 특화 요소(모델 요구사항, 신뢰도, 실패 처리)까지 포함하고 싶다면 이걸 써보세요.

### 📌 진행 방식

이 프롬프트는 **6단계 CoT(Chain of Thought)** 구조입니다. 두 가지 방식으로 쓸 수 있습니다:

**방식 A: 한 번에 넣기 (추천)**
- 아래 전체를 통째로 복사해서 AI에 붙여넣으세요
- AI가 모든 단계를 순서대로 실행합니다
- 결과가 한 번에 나옵니다 (길지만 완결적)

**방식 B: 단계별로 넣기 (더 깊은 분석)**
- 1-2단계를 먼저 넣고 결과를 확인하세요
- 결과가 만족스러우면 다음 단계를 이어서 넣으세요
- 중간에 수정이 필요하면 해당 단계를 수정 요청할 수 있습니다
- 시간이 더 걸리지만, 각 단계를 직접 검증할 수 있습니다

**`<!-- 💬 -->` 교육 주석이란?**
- 프롬프트 안에 `<!-- 💬 ... -->` 형태의 주석이 있습니다
- 이건 **여러분을 위한 설명**이지, AI에게 보내는 지시가 아닙니다
- AI에 붙여넣을 때 포함해도 되고 빼도 됩니다 (결과에 영향 없음)
- "왜 이렇게 설계했는가?"를 이해하면 더 좋은 프롬프트를 만들 수 있습니다

> 📖 더 자세한 사용법은 [프롬프트 사용 가이드](../meta/how-to-use.md)를 참고하세요.


### # 역할

AI PM, 데이터 과학팀 리더, ML 엔지니어, 제품 관리자

### # 목표

**AI 제품의 모든 차원을 명확하고 측정 가능하게 정의**

- 비즈니스 가치와 AI 기술 역량 연결
- 데이터 수집, 학습, 배포의 전체 생명주기 정의
- 성능 메트릭을 비즈니스 메트릭으로 번역
- 편향, 윤리, 규정 준수를 처음부터 설계
- 지속적 개선과 모니터링 계획 포함

### # AI PRD의 9가지 필수 섹션

#### Section 1: 비즈니스 케이스 & 용도 (Use Case)

<!-- 💬 [전문가의 눈] AI PRD가 실패하는 이유는 "비즈니스 케이스가 약할 때"입니다.
프로덕트팀: "모델 정확도 90%를 목표로"
경영진: "90%가 뭐하는 건데? 비즈니스에 뭐가 좋은데?"
따라서 비즈니스 목표(ROI, 매출, 비용절감)부터 시작한 후, 그것을 달성하려면 "정확도 85% 이상" 필요하다는 순서로 설명해야 합니다.
-->

**가장 중요합니다. 기술은 나중입니다.**

```
구조:

## 1. 비즈니스 목표
- 목표: [구체적 비즈니스 결과]
- 영향: [매출 증가? 비용 절감? 고객 만족?]
- 규모: [얼마나 많은 고객/거래에 영향?]

예시:
목표: 고객 이탈 예측
영향: 연 $2M 매출 손실 방지 (고객 이탈 1,000명)
규모: 활성 고객 50,000명 중 10% 대상

## 2. 현재 상태 (As-is)
- 프로세스: [현재 어떻게 하고 있나?]
- 문제점: [무엇이 문제인가?]
- 비용: [시간/돈으로 얼마나 드는가?]

예시:
프로세스: 분석팀이 수동으로 고객 데이터 분석
문제점: 분석에 3주 소요, 특정 신호 놓침
비용: 분석가 4명의 월급 + 손실 매출

## 3. 미래 상태 (To-be)
- 솔루션: [AI로 뭘 할 것인가?]
- 성과: [어떻게 개선되나?]
- ROI: [투자 대비 수익]

예시:
솔루션: AI가 자동으로 이탈 위험 고객 추출
성과: 3주 → 2시간, 정확도 85%, 대응률 60% → 80%
ROI: AI 구축 비용 $200K → 연 $2M 절감 (1년 내 회수)

## 4. 사용자 페르소나 & 시나리오

예시:
페르소나: 고객 성공 담당자 "김성수" (2년 경력)
시나리오: 매일 아침 "이탈 위험 고객 리스트" 대시보드 확인
→ 상위 10명을 우선 컨택 → 할인 쿠폰 제공 → 82% 잔존

이 시나리오가 없으면, "왜 이 정확도가 필요한가?"를 알 수 없습니다.
```

#### Section 2: AI 모델 정의

<!-- 💬 [논쟁 지점] 모델 정확도의 정의는 팀마다 다르다.
팀 A: "95% 정확도" (전체 샘플 중 맞춘 비율)
팀 B: "민감도 90%, 특이도 80%" (거짓 양성/음성 구분)
같은 95%도 다르게 해석될 수 있다. 따라서 "정밀도/재현율/F1" 세 가지를 명시해야 한다.
한국은 False Positive 비용이 높으므로 "정밀도"를 더 중시하는 경향.
-->

**"검은 상자"를 열어서 설명하세요.**

```
구조:

## 1. 작업 정의 (Task Definition)
- 유형: 분류(Classification) / 회귀(Regression) / 순위(Ranking) / 생성(Generative)
- 입력: 모델에 들어가는 데이터 (예: 고객 프로필, 구매 기록)
- 출력: 모델이 뱉는 결과 (예: "이탈 가능성 72%")
- 타겟: 예측하려는 것 (예: "30일 내 이탈 여부")

예시:
유형: Binary Classification (이탈/유지)
입력: {고객_나이, 결제_빈도, 마지막_접속, 고객센터_연락_빈도, ...}
출력: 이탈_가능성_점수 (0~100%)
타겟: 실제 이탈 여부 (관찰된 데이터에서)

<!-- 💬 왜 이 섹션이 중요한가?
"고객 이탈을 예측한다"는 말은 모호합니다.
"30일 내 이탈 여부를 이항 분류한다"는 구체적입니다.
이 구체성이 엔지니어, 데이터 과학가, PM이 같은 것을 보도록 합니다.
-->

## 2. 모델 선택
- 후보: [고려 중인 모델들]
- 선택: [선택한 모델과 이유]
- Trade-off: [성능 vs 해석 가능성 vs 속도]

예시:
후보: Logistic Regression (해석 쉬움), XGBoost (정확도 높음), 신경망 (복잡)
선택: XGBoost
이유: 85% 정확도 달성하면서도 Feature Importance로 설명 가능
Trade-off: 신경망보다 정확도 1% 낮지만, "왜 예측했나" 설명 가능

## 3. 성능 목표
- 정확도: [X% 이상]
- 정밀도 (Precision): [Y% 이상] ← False Positive 줄임
- 재현율 (Recall): [Z% 이상] ← False Negative 줄임
- F1 스코어: [값]

예시:
정확도: 85% (전체 예측 중 맞은 비율)
정밀도: 80% (이탈이라고 예측한 것 중 실제 이탈 비율)
재현율: 70% (실제 이탈한 고객 중 포착한 비율)
→ 의미: "100명에게 알림을 보내면 80명은 실제 위험, 70명은 미리 잡힘"

## 4. 개발 데이터셋
- 소스: [어디서 데이터를 얻을까?]
- 크기: [몇 개 샘플?]
- 시간 범위: [언제부터 언제까지?]
- 라벨 방식: [어떻게 "정답"을 만들까?]

예시:
소스: 회사 고객 데이터베이스
크기: 50,000명 고객 × 6개월 거래 데이터
시간 범위: 2023년 1월~6월 (학습), 7월~12월 (테스트)
라벨 방식: 실제 이탈 여부 (2024년 1월 기준)
```

#### Section 3: 데이터 아키텍처 & 파이프라인

<!-- 💬 [자기 검증] 데이터 파이프라인 설계 후 반드시 확인할 사항:
1. "데이터 누수(leakage)가 없는가?" → 미래 정보가 학습에 섞여있지 않은가?
2. "프라이버시를 보호했는가?" → PII(개인정보) 마스킹, 암호화?
3. "데이터 품질 게이트가 있는가?" → "정확도 85%에 못 미치면 모델 배포 금지"?
한국은 PIPA(개인정보보호법) 때문에 데이터 보호가 매우 엄격. 설계 단계부터 법무팀 검토 필수.
-->

**AI는 "모델"이 아니라 "데이터 파이프라인"입니다.**

```
구조:

## 1. 데이터 소스
- Source 1: [원본 데이터 위치]
- Source 2: [...]
- 업데이트 주기: [얼마나 자주 새 데이터를 가져올까?]
- 데이터 품질: [결측치, 이상치 처리]

예시:
Source 1: 고객 마스터 (OLTP DB)
Source 2: 거래 로그 (데이터 웨어하우스)
Source 3: 고객센터 티켓 (CRM)
업데이트: 매일 밤 자동 수집
품질: 결측치는 평균값으로 채움, 이상치는 통계적 검출

## 2. 데이터 처리 (ETL)
- 정제: [어떤 전처리?]
- 특성 공학: [어떤 새로운 변수 만들기?]
- 검증: [데이터 품질 확인]

예시:
정제:
  - 결측치: 평균/중앙값으로 보완
  - 이상치: IQR 방식으로 감지 및 제거
  - 중복: 거래 ID 기준 제거

특성 공학:
  - 구매_빈도 (월 평균)
  - 마지막_접속_일수 (오늘로부터 몇 일?)
  - 고객센터_접촉_횟수 (지난 3개월)

검증:
  - 결측치 5% 이하
  - Feature별 분포 확인
  - 데이터 누수(leakage) 검토

<!-- 💬 데이터 파이프라인이 왜 중요한가?
모델 정확도 85% 중:
- 모델 알고리즘: 40%
- 데이터 품질: 50%
- 특성 공학: 10%

즉, "좋은 데이터"가 "좋은 모델"보다 5배 중요합니다.
-->

## 3. 저장소 & 접근
- 학습 데이터: [어디 저장?]
- 모델: [어디 저장?]
- 예측 결과: [어디 저장?]
- 접근 제어: [누가 볼 수 있나?]

예시:
학습 데이터: S3 (AWS)
모델: Model Registry (MLflow)
예측: DynamoDB (빠른 조회용)
접근: IAM 권한 (PM은 읽기만, DS는 쓰기 가능)

## 4. 모니터링 & 재학습
- 모니터링: [뭘 추적할까?]
- 재학습 주기: [얼마나 자주 모델을 업데이트?]
- 성능 저하 감지: [어떤 경보?]

예시:
모니터링:
  - 일일: 예측 정확도, 신뢰도 분포
  - 주간: 그룹별 공정성 (성별, 나이대)
  - 월간: 데이터 드리프트 감지

재학습:
  - 주기: 매주 월요일 (새로운 7일 데이터)
  - 또는: 정확도가 82% 아래로 떨어질 때

성능 저가 감지:
  - 정확도 < 80% → 경고
  - 정확도 < 75% → 모델 롤백

#### Section 4: 성능 & 신뢰도 기준

<!-- 💬 [전문가의 눈] 신뢰도(Confidence) 메트릭을 반드시 포함해야 하는 이유:
모델이 "85% 정확도"라고 해도, 그 정확도가 일관된가?
어떤 타입의 입력은 95% 정확하지만, 어떤 타입은 50%일 수 있다.
따라서 "신뢰도가 낮은 예측은 사용자에게 경고"를 설계해야 한다.
모델 정확도 > 신뢰도 임계값 설정 > 신뢰도 낮은 결과는 자동 거절/전문가 판단 요청.
-->

```
구조:

## 1. 정확도 (Accuracy) - 전체 정확도

목표: 85%

의미: 100개의 예측 중 85개가 맞다

왜 85%인가?
- 비즈니스 영향도 (High) → 높은 정확도 필요
- False Positive 비용: 불필요한 컨택비 $100 × 20명 = $2,000
- False Negative 비용: 고객 이탈 손실 $2,000 × 15명 = $30,000
→ False Negative 비용이 15배 크므로, Recall을 높여야 함

## 2. 정밀도 (Precision) - 거짓 경보 줄이기

목표: 80%

의미: "이탈" 예측 중 80%가 실제 이탈

개선 이유:
- 지나치게 많은 컨택은 고객 불만 유발
- 정확한 타겟팅으로 비용 효율성

## 3. 재현율 (Recall) - 놓친 케이스 줄이기

목표: 70%

의미: 실제 이탈한 고객 중 70%를 미리 잡음

개선 이유:
- 남은 30%를 놓치면 $30K 손실
- 70%만 잡아도 충분한 ROI

## 4. 신뢰도 스코어 (Confidence Score)

- 정의: 모델이 "얼마나 확신하는가?"
- 임계값: 70% 이상만 자동 액션, 이하는 사람이 검토
- 거부율: 전체 예측의 15% 정도 수동 검토 (정상)

## 5. 공정성 메트릭 (Fairness)

테스트 그룹:
- 성별 (남/여): 성능 격차 < 5%
- 나이 (20~40 / 40~60 / 60+): 격차 < 5%
- 신규 vs 기존: 격차 < 5%

의미:
- 모델이 특정 집단을 차별하지 않는다
- 법적 의무 (차별금지법)
- 윤리적 책임
```

### 🇰🇷 한국 시장 AI PRD 특화 요구사항

```
한국 시장 AI 기능 PRD 작성 시 추가 섹션:

1. **개인정보보호법(PIPA) 준수 섹션**:
   - 데이터 수집 범위: 어떤 개인정보를 수집/이용하는가?
   - 사용자 동의: 명시적 동의 필요한가, 암묵적 동의로 가능한가?
   - 데이터 보관: 얼마나 오래 보관? (보관 기간 제한 필수)
   - 제3자 제공: 어떤 외부 회사에 데이터를 전달하는가?
   - 예시: "고객 거래 데이터는 모델 학습에만 사용, 마케팅 목적으로는 사용 금지"

2. **한국어 NLP 특성 고려**:
   - 형태소 분석: "은/는", "이/가" 등 한국어 문법 처리
   - 존댓말/반말: AI 응답의 톤앤매너 조정 (B2B는 존댓말, SNS는 캐주얼)
   - 초성/중성/종성: 자음/모음 분리 시 정확도 영향
   - 신조어: "오늘 핫하다", "꿀잼" 같은 신조어 학습 필요
   - 예시: "응답 생성 모델은 존댓말 톤으로, 정확도 기준 한국어 텍스트에서 85% 이상"

3. **한국 기업 의사결정 프로세스**:
   - PRD 작성 후 "기획 → 검토(법/보안) → 결재(임원)" 단계 필요
   - 법무팀 검토: PIPA, 통신비밀보호법, 정보통신망 이용촉진법
   - 보안팀 검토: 데이터 암호화, 접근 제어
   - 따라서 PRD는 "기술 PM용" 버전 + "경영진/법무용" 버전 2가지 필요
   - 예시: "이 기능은 개인정보보호법 제2조 수집 범위에 해당하며, 명시적 동의 필수"

4. **한국 AI 규제 요구사항**:
   - AI기본법(2024): 고위험 AI 시스템은 영향 평가 필수
   - 고위험 판단: "신용/채용/법 집행/교육" 관련 AI → 정부 신고 필요
   - 대응: PRD에 "영향 평가 계획" 섹션 추가
   - 예시: "고객 이탈 예측은 마케팅 목적이므로 저위험, 신고 불필요. 다만 월간 모니터링 리포트 필수"

5. **한국 시장 테스트 및 검증**:
   - A/B 테스트 전 법무팀 동의 필수 (개인정보 사용 범위 확인)
   - 베타 테스트: 실명 기반 한국 사용자 샘플 필수 (영어 테스트 데이터 사용 X)
   - 검증: 한국 고객센터 팀과 협업 (실제 사용 피드백)
```

PRD 작성은 기능마다 반복됩니다. 자동화하면 일관된 품질의 PRD를 빠르게 생성할 수 있습니다.

**전제조건:**
- Python 3.8+ 설치
- 필요한 패키지 설치 (코드 상단의 import 참고)
- Anthropic API 키 발급 → 환경변수 설정: `export ANTHROPIC_API_KEY=your_key`

> 📖 더 자세한 설정 방법은 [프롬프트 사용 가이드](../meta/how-to-use.md)를 참고하세요.

AI 기능 PRD 자동 생성 도구 (FLAGSHIP MVP)
Claude API를 사용하여 비즈니스 요구사항에서 전체 AI PRD 생성

주요 기능:

import anthropic

from typing import Optional, Dict, List
from dataclasses import dataclass, asdict
from enum import Enum


class AITaskType(Enum):
    """AI 작업 타입"""
    CLASSIFICATION = "classification"
    REGRESSION = "regression"
    RANKING = "ranking"
    GENERATIVE = "generative"
    FORECASTING = "forecasting"


class BusinessImpact(Enum):

    HIGH = "high"      # 정확도 90%+ 필요
    MEDIUM = "medium"  # 정확도 80-90%
    LOW = "low"        # 정확도 70-80%


@dataclass
class AIProductRequirements:
    """AI 제품 요구사항"""
    product_name: str
    use_case: str  # 예: "고객 이탈 예측"
    ai_task_type: AITaskType
    business_impact: BusinessImpact
    target_users: str  # 예: "고객 성공 담당자"
    data_sources: List[str]  # 데이터 소스 목록
    constraints: Optional[str] = None  # 규정, 비용 등


def generate_ai_prd(requirements: AIProductRequirements) -> Dict:
    """
    AI 기능의 전체 PRD 생성

    1. 비즈니스 케이스
    2. AI 모델 정의
    3. 데이터 아키텍처
    4. 성능 기준
    5. 시스템 아키텍처
    6. 편향 & 윤리 & 규정
    7. 팀 역할
    8. 피드백 루프
    9. 로드맵

6. **편향 & 윤리 & 규정**

   - 완화 전략
   - 한국 규정 준수 (개인정보보호법, AI 윤리)
   - 투명성 & 설명 가능성

7. **팀 역할 & 책임**
   - 각 역할 (PM, DS, Engineer, Compliance)

   - 의사결정 구조

8. **피드백 루프 & 지속적 개선**
   - 피드백 수집 방식
   - 성능 분석 (주간/월간)

   - 배포 프로세스
   - 성공/실패 사례

9. **로드맵 & 마일스톤**

   - Phase 2-5 (각 2주)
   - Phase 6 정상화

### Input: AI 기능 요구사항

```
제품: 고객 이탈 예측 시스템
사용 사례: 연 $2M 손실을 방지하기 위해 고객 이탈 위험을 미리 감지
AI 작업: 이항 분류 (이탈/유지)
비즈니스 영향: High (재무적 큰 손실)
대상: 고객 성공 팀 (20명)
데이터: 고객 마스터, 거래, 지원 티켓, 결제
제약: ISMS, GDPR, 한국 개인정보보호법
```

### Output: 생성된 AI PRD (9개 섹션, 각 500자+)

```markdown
# 고객 이탈 예측 시스템 PRD

---

## 1. 비즈니스 케이스

### 비즈니스 목표
연간 $2M의 고객 이탈 손실을 20-30% 줄인다 (연 $400K~600K 절감).
이를 위해 이탈 위험 고객을 30일 전에 자동 감지하고,
고객 성공 팀이 우선 개입할 수 있도록 한다.

### 현재 상태 (As-is)
**프로세스:**
- 분석팀이 월 1회 고객 데이터 수동 분석
- 각 분석가가 엑셀로 고객 건강도 점수 계산
- 3주 소요 (지연 발생)
- 신호를 놓치는 경우 많음 (휴리스틱 기반)

**문제점:**
- 반응 시간이 느림 (3주 → 이미 이탈한 고객 많음)
- 개인별 분석 기준이 다름 (일관성 없음)
- 특정 패턴 포착 불가 (반복되는 신호 놓침)
- 리소스 낭비 (분석가 4명의 월급 ~$100K)

**비용:**
- 분석가 급여: 월 $100K × 12 = $1.2M/년
- 기회 손실: 이탈하지 않았을 고객 = $2M/년
- 총 비용: $3.2M/년

### 미래 상태 (To-be)
**솔루션:**
AI 모델이 매일 자동으로 모든 고객 50,000명의 이탈 위험도를 계산.
점수가 높은 상위 100명을 고객 성공팀에게 자동 알림.
팀이 우선 개입 (할인쿠폰, 1:1 컨택, 업그레이드 제안).

**개선사항:**
- 대응 시간: 3주 → 하루 (30배 빠름)
- 정확도: 70% (휴리스틱) → 85% (AI)
- 일관성: 자동화로 100%
- 비용: 분석가 급여 $1.2M → AI 운영 $200K (약 6배 절감)

**ROI:**
- 투자: AI 구축 + 운영 $300K (1년)
- 수익: 고객 유지로 인한 매출 $400K~600K (절감)
- 순이익: $100K~300K (1년 내 회수)

### 사용자 페르소나

**페르소나 1: 고객 성공 담당자 "박준호"** (2년 경력)
- 역할: 50명 고객 담당, 분기별 이탈 목표 5% 이하
- 시나리오:
  아침 9시 대시보드 접속 → "이탈 위험 고객 TOP 10" 확인

## 2. AI 모델 정의

### 작업 정의 (Task Definition)

**유형:** 이항 분류 (Binary Classification)

**입력 데이터 (Features):**
- 고객_연령, 가입일수, 결제금액, 결제빈도
- 마지막_접속_일수, 로그인_빈도_28일
- 고객센터_접촉_횟수_90일, 반환/환불_여부
- 가격인상_이후_행동, 경쟁사_가입_신호

**출력:**
- 이탈_가능성_점수: 0~100 (예: 72%)
- 신뢰도_스코어: 0~100 (예: 85% "확신함")
- 이탈_예상_시점: "30일 이내" or "6개월 이내"

**타겟 (정답):**
실제로 30일 이내 이탈한 고객 여부
(2023년 데이터 기반, 2024년 1월까지 관찰)

### 모델 후보 & 선택

**후보:**
1. **Logistic Regression** - 장점: 해석 쉬움 / 단점: 정확도 낮음 (70%)
2. **XGBoost** - 장점: 정확도 높음 (85%) / 단점: 복잡함
3. **신경망 (NN)** - 장점: 최고 정확도 (87%) / 단점: 매우 복잡, 해석 불가

**선택: XGBoost**

**이유:**
- 85% 정확도 목표 달성 가능
- Feature Importance로 "왜 이탈 위험인지" 설명 가능
- 학습/예측 빠름 (수초)
- 한국 규정상 "설명 가능성" 필수 → XGBoost가 최적

### 성능 목표

| 메트릭 | 목표 | 비즈니스 의미 |
|--------|------|-------------|
| **정확도 (Accuracy)** | 85% | 100개 예측 중 85개 맞음 |
| **정밀도 (Precision)** | 80% | 알림 받은 100명 중 80명이 실제 위험 |
| **재현율 (Recall)** | 70% | 실제 이탈한 100명 중 70명을 미리 잡음 |
| **F1 스코어** | 0.75 | 정밀도와 재현율 균형 |

**정확도가 85%인 이유:**
- False Positive (거짓 경보) 비용: 불필요한 컨택 $20 × 20명 = $400
- False Negative (놓친 고객) 비용: 이탈 손실 $2,000 × 15명 = $30,000
→ False Negative가 75배 비싸므로, **정밀도(80%)와 재현율(70%)의 균형** 필요

## 4. 성능 & 신뢰도 기준

### 정확도 (Accuracy): 85%

**정의:** 모든 예측 중 85%가 맞다

**계산:** (TP + TN) / (TP + TN + FP + FN)

**테스트 방법:**
- Test set: 2024년 고객 10,000명 (학습에 미사용)
- Ground truth: 실제 이탈 여부 (2024년 2월 관찰)
- 메트릭: Accuracy, Precision, Recall, F1

### 정밀도 (Precision): 80%

**정의:** "이탈 위험"이라고 예측한 것 중 80%가 실제 이탈

**비즈니스 의미:**
- 100명에게 "위험" 알림 → 80명이 실제 위험
- 20명은 거짓 경보 (불필요한 컨택)

**개선 동기:**
- 거짓 경보가 많으면 팀이 피로 (Boy who cried wolf)
- 고객도 불필요한 컨택에 피로

### 재현율 (Recall): 70%

**정의:** 실제 이탈한 고객 중 70%를 미리 잡음

**비즈니스 의미:**
- 실제 100명이 이탈 → 70명을 미리 감지
- 30명은 놓침

**개선 동기:**
- 놓친 30명 = $60K 손실
- 하지만 Recall 80%로 올리려면 모델 복잡화 필요
- 비용-편익 분석: 70%가 최적

### 신뢰도 스코어 & 임계값

```
모델 출력: {이탈_점수: 72%, 신뢰도: 85%}

임계값 정책:
- 신뢰도 80% 이상 → 자동 고위험 알림 (우선도 높음)
- 신뢰도 60~80% → 보조 정보로 제시 (우선도 중간)
- 신뢰도 60% 이하 → 표시 안 함 (신뢰 불충분)

효과:
- 신뢰도 높은 것만 → 팀이 신뢰
- 신뢰도 낮은 것은 → 수동 검토 또는 무시
- 거짓 경보 비율 유지

거부율 (Abstention Rate):
- 목표: 전체 예측의 15% 정도를 수동 검토로 전환
- 의미: "자동 알림 85개 + 수동 검토 15개" = 100명/일
```

### 공정성 메트릭 (Fairness)

```
테스트 그룹별 성능 격차:

그룹 A: 신규 고객 (가입 < 1년)
정확도: 82% (목표 85%)
→ 차이: -3% (허용 범위)
→ 이유: 신규 고객 데이터 부족
→ 행동: 신규 고객 데이터 수집 강화

그룹 B: 고액 고객 (MRR > $10K)
정확도: 88% (목표 85%)
→ 차이: +3% (허용 범위)
→ 의미: 고액 고객은 예측하기 쉬움
→ 위험 없음

그룹 C: 여성 고객
정확도: 84% (목표 85%)
재현율: 68% (목표 70%)
→ 차이: -2%, -2% (허용 범위)
→ 행동: 모니터링 계속

공정성 기준:
✓ 그룹별 정확도 차이 < 5%
✓ 그룹별 정밀도 차이 < 5%
✓ 그룹별 재현율 차이 < 5%

의미:
- 어떤 그룹도 차별받지 않는다
- 법적 의무 (차별금지법)
- 신뢰 구축
```

### 레이턴시 (Response Time)

```
목표:
- 평균 추론 시간: 50ms
- p95 레이턴시: 150ms
- p99 레이턴시: 300ms

의미:
- 50,000명 데일리 스캔: 50ms × 50K = 2,500초 = 42분 (야간 실행 가능)
- 실시간 개별 조회: 300ms 이내 (사용자 인내)

최적화:
- 모델: XGBoost (신경망보다 10배 빠름)
- 배치 처리: 50명씩 한 번에 (벡터화)
- 캐싱: 자주 조회되는 고객은 캐시에서 즉시 반환
```

## 6. 편향 & 윤리 & 규정

### 가능한 편향

```
1. 데이터 편향 (Data Bias)

신규 고객 vs 기존 고객:
- 신규: 거래 기록 < 3개월 (특성 부족)
- 기존: 거래 기록 > 1년 (특성 풍부)
→ 모델이 신규 고객을 예측하기 어려움
→ 신규 고객의 정확도: 82% (기존 85% 대비)

해결:
- 신규 고객 데이터 집중 수집 (3개월 특성)
- 신규 전용 모델 개발 (별도)

고액 vs 저액 고객:
- 고액: 거래 빈도 높음 (신호 명확)
- 저액: 거래 빈도 낮음 (신호 모호)
→ 모델이 고액 고객 예측 더 정확 (88% vs 78%)

해결:
- 그룹별 모델 또는 가중치 조정

2. 알고리즘 편향 (Algorithmic Bias)

성별 편향:
- 남성 정확도: 86%
- 여성 정확도: 83%
→ 차이: 3% (허용 범위 5% 내)

해결:
- 여성 고객 데이터 오버샘플링
- 여성 특화 특성 추가

3. 사용 편향 (Usage Bias)

팀이 모델 점수를 맹목적으로 신뢰하는 문제:
- 예: "모델이 72%라고 했으니까 이 고객은 이탈해"
→ 실제로는 70% 신뢰도 (30% 오류 가능성)

해결:
- 신뢰도 스코어 함께 표시
- 팀 교육: "70% 신뢰도 = 10번 중 3번은 틀림"
- 최종 결정은 사람이 하도록 (모델은 보조 도구)
```

### 완화 전략

```
1. 데이터 편향 → 균형 있는 학습 데이터

- 신규 고객 데이터 수집 강화
- 저액 고객 샘플 크기 증가
- 계층화 샘플링 (Stratified Sampling)

2. 알고리즘 편향 → 공정성 제약 추가

- 그룹별 정확도 동일: |Accuracy_M - Accuracy_F| < 5%
- Fairness-aware 학습: 모델 학습 시 공정성 제약
- 예: XGBoost + Fairness constraint

3. 사용 편향 → 투명성 & 설명 가능성

- Feature Importance: "왜 이 고객이 위험?"
- 상위 3가지 이유 표시
  * "마지막 접속 15일전" (가장 중요)
  * "최근 활동 50% 감소" (두번째)
  * "고객센터 불만" (세번째)
- SHAP (Shapley) 값으로 개별 설명
```

### 한국 규정 준수

```
1. 개인정보보호법

의무사항:
- "자동 결정에 관한 설명 요청" 수용
- 고객이 요청 시: "왜 이 고객이 이탈 위험인가?" 설명
- 최소 3가지 이유 제시

대응:
- Feature Importance로 자동 설명 생성
- 1주일 내 고객 요청 응답 프로세스
- 설명 불만 시 재검토 프로세스

2. AI 윤리 가이드 (미래부)

요구사항:
- 투명성: 모델 작동 방식 공개
- 공정성: 그룹별 차별 없음
- 설명 가능성: 예측 이유 설명 가능
- 인간 개입: 최종 결정은 사람

대응:
- 모든 항목 만족

3. 금융감독 (신용도 영향 시)

만약 이 모델이 신용점수에 영향을 미친다면:
- 모델 위험 평가 필수
- 정기 감시 보고
- 감사 흔적 (Audit trail)

대응:
- 현재 모델은 "마케팅 제안"이므로 금융감독 미적용
- 향후 신용점수 연결 시 재검토

### 모니터링 대시보드

```
1. 일일 모니터링

지표:
- 정확도 (어제: 85.2%, 7일 평균: 85.1%)
- 신뢰도 분포 (80% 이상: 72%, 70~80%: 20%, 미만: 8%)
- 에러 케이스 (예: API 오류 5건, 모델 오류 2건)

차트:
- 정확도 시계열 (7일)
- 신뢰도 히스토그램
- 에러 분포

임계값 & 알림:
- 정확도 < 82%  → Red (즉시 조사)
- 에러 > 5% → Yellow (팀 검토)

2. 주간 모니터링

지표:
- 그룹별 정확도 (남: 86%, 여: 83%, 신규: 82%)
- 그룹별 False Positive (남: 18%, 여: 22%, 신규: 25%)
- 데이터 드리프트 (IQR 기반 변화 감지)

테이블:
| 그룹 | 정확도 | 정밀도 | 재현율 | 상태 |
|------|--------|--------|--------|------|
| 남성 | 86% | 82% | 72% | ✅ |
| 여성 | 83% | 80% | 68% | ⚠️ |
| 신규 | 82% | 78% | 65% | ⚠️ |

임계값:
- 정확도 격차 > 5% → Yellow (조사 필요)
- False Positive 격차 > 5% → Yellow
- 데이터 드리프트 감지 → 모델 성능 확인

3. 월간 모니터링

항목:
- 모델 성능 추이 (지난 3개월)
- 편향 감시 결과
- 비즈니스 영향 (실제 이탈율 개선%)
- 규정 준수 점검

결과:
- 진행 중 요약
- 다음 달 액션 아이템
```

## 8. 피드백 루프 & 지속적 개선

### 피드백 수집

```
1. 자동 피드백 (매일)

모델 예측과 실제 결과 비교:
- 모델: "이 고객 72% 이탈 위험"
- 실제: 30일 후 고객 실제 이탈 여부 관찰
- 결과: 정확/오류 기록

소스:
- 고객 이탈 여부 (CRM)
- 고객 활동 재개 여부 (로그인)
- 고객 문의 (해결됨?)

저장: DynamoDB (매일 업데이트)

2. 사용자 피드백 (선택적)

팀이 모델을 사용하면서 의견 제시:
- "이 고객은 위험하지 않아" → False Positive 표시
- "이 고객을 미리 잡아서 도움 됐어" → True Positive 확인
- "왜 이 고객이 위험인지 이해 안 됨" → 설명 부족

UI:
```
[예측 결과] 72% 위험

[버튼]
👍 맞다 (이탈함)
👎 틀렸다 (유지함)
❓ 왜? (설명 부족)
```

저장: 피드백 로그 (주간 수집)

3. 비즈니스 피드백 (월간)

실제 비즈니스 성과:
- 모델이 추천한 고객 몇 명이 실제로 유지되었나?
- 비용 절감액은?
- 팀 워크로드 변화?

소스:
- CRM (실제 이탈율)
- 재무 (비용 절감액)
- 팀 설문 (만족도)

### 성능 분석 (주간)

```
매주 금요일 분석:

1. 정확도 분석

지난 주 예측 1,000건:
- 정확: 850건 (85%)
- 오류: 150건 (15%)

오류 분석:
- False Positive (거짓 경보): 100건
  → 왜? 마지막 접속이 14일인데 실제로는 유지
  → 임계값 조정? 특성 변경?

- False Negative (놓친 것): 50건
  → 왜? 모델이 30%라고 했는데 실제로 이탈
  → 데이터 품질 문제? 특성 부족?

2. 그룹별 분석

성별:
- 남성: 정확도 86%
- 여성: 정확도 83%
→ 여성 그룹이 3% 낮음 (임계값 5% 내)
→ 다음 월: 여성 고객 데이터 수집 강화

연령대:
- 20~40: 정확도 86%
- 40~60: 정확도 84%
- 60+: 정확도 81%
→ 고연령대 정확도 낮음
→ 고연령대 특성 추가 필요?

3. 에러 케이스 상세 분석

오류 패턴:
- 패턴 1: 최근 대량 거래 후 갑자기 조용해짐
  → 기술적 문제? 납기 완료?
  → "활동 패턴 변화" 특성 추가

- 패턴 2: 신규 고객 (가입 < 3개월)
  → 데이터 부족
  → 신규 고객 전용 모델 개발?

### 모델 재학습 (월 1회)

```
프로세스 (매월 첫 주 월요일):

Step 1: 데이터 수집 (새로운 1개월 데이터)
  - 고객 50,000명의 최신 거래, 활동, 지원 데이터
  - 라벨: 지난 달에 이탈한 고객

Step 2: 특성 계산
  - 기존과 동일한 방식으로 15개 특성 계산
  - 새로운 특성 추가? (이전 달 피드백 기반)

Step 3: 모델 훈련
  - 지난 12개월 데이터로 XGBoost 재학습
  - 하이퍼파라미터 자동 튜닝

Step 4: 테스트 셋 평가
  - 신 모델 정확도: 85.5%
  - 구 모델 정확도: 85.2%
  - 개선: +0.3% (통계적으로 유의하지 않음)

Step 5: A/B 테스트 (옵션)
  - 신 모델이 구 모델보다 명확히 더 나으면
  - 1주일 동안 10% 트래픽에서 A/B 테스트
  - 결과: 신 모델의 실제 이탈 방지율 vs 구 모델

Step 6: 배포 결정
  - 성능 동일하면: 배포 안 함 (현재 모델 유지)
  - 성능 향상하면: Canary 배포 (10% → 100%)
  - 성능 악화하면: 조사 (데이터 드리프트? 특성 문제?)

### 지속적 개선 아이디어

```
분기별 우선순위:

Q1: 데이터 품질
- 신규 고객 데이터 수집 강화
- 여성 고객 특성 강화
- 고연령대 특성 추가

Q2: 모델 개선
- 신규 고객 전용 모델 개발?
- LightGBM vs XGBoost 비교
- 하이퍼파라미터 최적화

Q3: 사용성 개선
- 팀 피드백 기반 UI 개선
- 설명 가능성 강화 (SHAP)
- 모바일 앱 지원

Q4: 확대
- 다른 사용 사례? (업셀 가능성?)
- 신용도 영향? (규제 검토 필요)
- 타사 대리점에게 제공?
```

  * 15개 특성 설계 및 계산

  * XGBoost 기본 모델 개발
