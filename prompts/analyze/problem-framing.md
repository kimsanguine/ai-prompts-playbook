# 🎯 문제 정의 프레이밍 (Problem Framing)

> 예상 소요: 5분 (Quick) / 25분 (Deep) / 35분+ (Build)
> 워크플로우: ← `customer-journey-mapping.md` ← `persona-profiling.md` → `jobs-to-be-done.md`

---

## 🎯 이 프롬프트가 해결하는 문제

"뭔가 이상한데, 뭔지 모르겠다"는 상태로 기능 개발을 시작합니다. 문제를 제대로 정의하지 않으면, 해결책도 엉뚱해집니다. "빠른 협업 도구"라고 생각했는데 실은 고객의 진짜 문제는 "팀이 같은 정보를 봐야 한다"일 수 있습니다. 이 프롬프트는 **"우리가 풀어야 할 진짜 문제가 뭔가?"**를 명확히 합니다.

---

## 📥 이전 프롬프트에서 받는 입력

> 워크플로우의 시작점입니다. 모든 분석의 출발점으로, 이후 JTBD, 페르소나, 기능 우선순위로 연결됩니다.

입력이 필요 없습니다. **해결해야 할 문제 상황, 관찰된 증상, 이해관계자의 요청**을 직접 입력하세요.

---

## ⚡ Layer 1: Quick Start

> **이 레이어가 하는 일: 전문가의 눈 빌리기**
> 전문가가 이 분야를 볼 때 쓰는 프레이밍을 빌려줍니다. AI를 처음 쓰는 분, 비개발자, 학생 — 5분이면 제품/서비스의 핵심 문제를 정의할 수 있습니다.

### 📌 사용법

1. **ChatGPT, Claude, Gemini 중 아무 AI 챗봇**을 열고 새 대화를 시작하세요
2. 아래 프롬프트를 **통째로 복사**해서 AI에 붙여넣으세요
3. `{중괄호}` 안의 내용만 여러분의 상황에 맞게 바꾸세요
4. Enter를 누르면 끝!

> 📖 더 자세한 사용법은 [프롬프트 사용 가이드](../meta/how-to-use.md)를 참고하세요.


```
다음 중 현재 상황을 고르세요:

A. "우리 제품을 못 씀" → 장벽/마찰점 찾기
B. "만족도가 떨어짐" → 기대 vs 현실 갭 찾기
C. "경쟁사에 진다" → 우리가 못 하는 게 뭔지 찾기
D. "수익성이 안 나온다" → 비즈니스 모델 이슈 찾기

선택한 상황에 대해:
1. "우리 고객이 지금 느끼는 가장 큰 고통은?" (구체적으로)
2. "우리가 그걸 못 해결하는 이유는?" (기능? 비용? 시간?)
3. "이걸 해결하면 뭐가 바뀌는가?" (측정 가능한 결과)

예:
A. 장벽: "가입 프로세스가 5단계 걸려서 포기"
B. 만족도: "기능은 좋은데 응답이 30초 걸려서 답답"
C. 경쟁사: "경쟁사는 한글 지원인데 우리는 영어만"
D. 수익성: "초기 고객 획득은 되는데 리텐션이 30%"
```

**💡 Quick Tips:**
- "문제"를 추상적으로 말하지 말기 (X: "UX가 복잡", O: "가입에 5단계, 인증 메일 기다림, 초기 설정 15분")
- 정량화하기 (X: "느리다", O: "첫 응답까지 평균 28초")
- 고객 말로 (X: "컨버전 지표 떨어짐", O: "고객이 말함: '설정이 너무 복잡해서 포기했어'")


### ⚠️ 이런 경우는요?

| 상황 | 대처법 |
|------|--------|
| 문제가 너무 추상적 | 구체적 메트릭을 추가하세요. "UX가 복잡" → "가입에 5단계, 평균 15분 소요" |
| 여러 문제가 섞여 있음 | 가장 영향이 큰 문제 1개만 먼저 정의하세요. 나머지는 따로 분석 |
| 데이터가 없음 | "고객 인터뷰 3건 + 지원 티켓 패턴"만으로도 시작 가능합니다 |
| 팀마다 문제 인식이 다름 | 이 프롬프트 결과를 공유해서 팀 정렬(alignment)에 활용하세요 |

## 🧠 Layer 2: Deep Dive

> **이 레이어는 누구를 위한 건가요?**
> PM, 기획자, 마케터 — 문제 정의를 전문가 수준으로 수행하고 싶은 분입니다.
Layer 1의 결과물이 "표면적"이라고 느꼈다면 이걸 써보세요.

### 📌 진행 방식

이 프롬프트는 **4단계 CoT(Chain of Thought)** 구조입니다. 두 가지 방식으로 쓸 수 있습니다:

**방식 A: 한 번에 넣기 (추천)**
- 아래 전체를 통째로 복사해서 AI에 붙여넣으세요
- AI가 모든 단계를 순서대로 실행합니다
- 결과가 한 번에 나옵니다 (길지만 완결적)

**방식 B: 단계별로 넣기 (더 깊은 분석)**
- 1-2단계를 먼저 넣고 결과를 확인하세요
- 결과가 만족스러우면 다음 단계를 이어서 넣으세요
- 중간에 수정이 필요하면 해당 단계를 수정 요청할 수 있습니다
- 시간이 더 걸리지만, 각 단계를 직접 검증할 수 있습니다

**`<!-- 💬 -->` 교육 주석이란?**
- 프롬프트 안에 `<!-- 💬 ... -->` 형태의 주석이 있습니다
- 이건 **여러분을 위한 설명**이지, AI에게 보내는 지시가 아닙니다
- AI에 붙여넣을 때 포함해도 되고 빼도 됩니다 (결과에 영향 없음)
- "왜 이렇게 설계했는가?"를 이해하면 더 좋은 프롬프트를 만들 수 있습니다

> 📖 더 자세한 사용법은 [프롬프트 사용 가이드](../meta/how-to-use.md)를 참고하세요.


```
# 역할: 문제 재정의 전문가 (Problem Reframing Expert)
# 목표: "우리가 푸는 문제"와 "우리가 풀어야 할 진짜 문제" 구분하기

# Reasoning Strategy: 4단계 문제 정의
<!-- 💬 왜 4단계인가?
     대부분 팀은 1단계 (증상 인식)에서만 머뭅니다.
     "느리다" → 그래서 뭐? (단계 2) → 그 이유가? (단계 3) → 그래서 우리가 뭘 해야? (단계 4)

     각 단계를 깊게 파면: 잘못된 방향의 40% 줄어듭니다. -->

1단계: 증상(Symptom) 인식
   <!-- 💬 "문제가 있다"는 증상만 봅니다. 아직 원인은 몰라요 -->

   현상 (관찰 가능한 사실):
   - 메트릭: "전환율 15% → 12%로 떨어짐 (지난 분기)"
   - 고객 피드백: "사용하다가 포기했어요"
   - 팀 관찰: "신규 가입자가 2주 내 이탈"
   - 영업팀: "데모 중 고객이 자주 질문: '설정이 어디 있어?'"

   → 증상: "뭔가 안 좋아지고 있다"

2단계: 문제(Problem) 정의 — 증상의 원인
   <!-- 💬 증상 아래의 실제 문제를 파내야 합니다.
        "느린 것" (증상) ← "어디가 느린가?" (문제) ← "왜 느린가?" (근본 원인)

        같은 증상도 원인은 다릅니다:
        - "가입을 포기함" ← 가입이 복잡함? 느림? 신뢰가 없어서?
   -->

   "왜인가?" 3단계 깊이

   Level 1: 넓은 문제 명제
   - "신규 사용자 리텐션이 낮다"
   - "온보딩 경험이 좋지 않다"

   Level 2: 구체화
   - "가입 후 7일 내 이탈율이 70%"
   - "첫 로그인 후 2분 내에 왜 떠나는가?"

   Level 3: 깊은 원인 파악 (고객 인터뷰)
   - "홈페이지 가입 과정이 이메일 인증 → 결제 입력 → 초기 설정 3단계"
   - "각 단계에서 5-10분 → 총 20분 소요"
   - "대기시간: 이메일 인증 기다림 (평균 2분)"
   - "고객 말: '너무 길어서 나중에 하려고 했는데 잊어버렸어요'"

   → 실제 문제: "초기 설정 프로세스가 길어서 고객이 중단함"

3단계: 문제의 "범위" 정의 (Who / When / Where)
   <!-- 💬 모든 사용자가 같은 문제를 겪나요? 아니면 특정 그룹만?

        예:
        - "모든 신규 사용자?" X → "첫 방문한 모바일 사용자만?"
        - "항상?" X → "피크 타임(9-10시)에만 느림?"
        - "우리 기능?" X → "경쟁사 A 대비?"

        범위를 좁히면: 우선순위가 명확해집니다.
   -->

   - **Who** (누가): 모든 신규 가입자 / 특정 국가 / 특정 직업 / 모바일 사용자?
   - **When** (언제): 항상 / 특정 시간 / 특정 요일 / 계절?
   - **Where** (어디서): 어느 페이지 / 어느 기능 / 어느 디바이스?
   - **How Often** (얼마나): 항상 / 70% 사용자 / 일부만?

   예:
   - Who: 신규 가입자 (기존 유저는 지나감) — 범위 좁아짐
   - When: 첫 48시간 내 (이후 자동화로 진행)
   - Where: 온보딩 플로우 (결제 후 초기 설정 단계)
   - How Often: 신규 사용자 70% (매우 일반적)

4단계: 문제의 영향(Impact) & 우선순위
   <!-- 💬 문제가 크면 우선순위가 높습니다.
        하지만 "크다"를 정량화해야 합니다.

        방정식: 영향도 = (영향받는 사용자 수) × (심각성) × (빈도)
   -->

   - **영향받는 규모**: 신규 사용자 월 ~500명 × 70% 이탈 = 월 350명 손실
   - **비즈니스 영향**: 350명 × 월 평균 LTV $50 = 월 $17,500 손실
   - **심각성**: "완전히 못 씀" vs "불편함"? → 완전히 못 쓰는 것 (심각도 HIGH)
   - **빈도**: "항상" vs "가끔"? → 항상 발생 (빈도 VERY HIGH)

   → 우선순위 점수: (500 × 0.7) × (심각도 10) × (빈도 1.0) = 3,500 포인트 (매우 높음)

5단계: "우리가 푸는 것" vs "우리가 풀어야 할 것"
   <!-- 💬 이게 핵심입니다!

        "문제의 크기"가 정해졌으면, 이제 "우리 솔루션의 적절성"을 판단해야 합니다.

        예:
        - 문제: "이메일 인증이 느림"
        - 우리 솔루션: "가입 단계를 3→2로 줄이기"
        - 실제로 도움? → 아니면 절반만 도움 (이메일 기다림 2분은 여전함)

        더 나은 솔루션:
        - "이메일 인증 -> SMS 즉시 인증으로" (2분 → 0초)
        - 또는 "소셜 로그인 (Google/GitHub) 추가" (모든 인증 스킵)
   -->

   현재 우리 솔루션안:
   - [방안 1]: _____
   - [방안 2]: _____

   각 솔루션이 문제의 몇 %를 해결하나?
   - 방안 1: __% (이유: ____)
   - 방안 2: __% (이유: ____)

   → 가장 효과적인 솔루션 선택 + 다른 대안도 파악

6단계: 최종 문제 명제 (Problem Statement) 작성
   <!-- 💬 명제가 좋으면 팀 전체가 같은 방향으로 갑니다 -->

   형식:
   ```
   [대상 사용자]는 [상황]에서
   [문제/고통]을 겪고 있습니다.

   현재 [현재 방식]으로 대응 중이지만,
   [왜 부족한가]에서 실패합니다.

   우리는 [우리 솔루션]으로 [구체적 결과]를 달성하려 합니다.

   성공 지표: [측정 가능한 숫자]
   ```

   예:
   ```
   신규 가입 사용자는 온보딩 프로세스에서
   길고 복잡한 설정 과정을 겪고 있습니다.

   현재 이메일 인증 → 결제 입력 → 초기 설정 3단계 (총 20분)로 대응 중이지만,
   대기 시간(이메일 2분) + 인지적 부담이 커서 70%가 포기합니다.

   우리는 소셜 로그인 + 스킵 가능한 초기 설정으로
   완료 시간을 20분 → 3분으로 단축하고자 합니다.

   성공 지표: 신규 사용자 7일 리텐션 70% → 85% (15%p 개선)
   ```

7단계: 작성한 문제 명제를 검토하세요.
   <!-- 💬 자기검증:
        [ ] 문제가 구체적인가? (일반화하지 않았나?)
        [ ] 수치화했나? (영향도, 범위)
        [ ] 고객 목소리가 들렸나? (추측이 아닌 피드백)
        [ ] 우리 솔루션이 정말 문제를 해결하나? (50% 해결? 100%?)
        [ ] 다른 문제는 없나? (이것만 풀면 괜찮나?)
   -->

# 컨텍스트와 데이터
## 상황: {현재 도전 과제}
## 증상들:
   - 메트릭: {수치}
   - 고객 피드백: {인터뷰 or 설문}
   - 팀 관찰: {영업/CS/엔지니어 피드백}

## 이미 수집한 데이터:
   - 페르소나: {페르소나명}
   - 고객 여정: {병목점}
   - JTBD: {고객이 하려는 일}
```

**💡 Deep Tips:**

- **데이터 수집:**
  - 고객 인터뷰 (최소 5명, "왜 포기했나?"에 집중)
  - 스크린 레코딩 (사용자가 실제로 어디서 막혔나?)
  - GA 퍼널 분석 (어디서 이탈 가장 많은가?)
  - Zendesk/Intercom 데이터 (지원 요청 패턴)
  - NPS/만족도 설문 (특히 낮은 점수 이유)
  - 이탈한 고객 인터뷰 (취소 이유)

- **AI 제품이라면:**
  - "AI 결과가 정확하지 않아서 못 씀" vs "결과는 맞는데 너무 느려서" (근본 원인 다름)
  - "할루시네이션 때문에 신뢰 못 함" (감정적 문제)
  - "데이터 프라이버시 때문에 못 씀" (규제 문제)
  - "비용이 너무 비싸서" (경제 문제)

- **한국 SaaS 특화:**
  - "한글 지원 안 돼서" (매우 빈번한 이유)
  - "한국 고객 지원이 없어서" (시간대 문제)
  - "결제 방법 제한" (신용카드/계좌이체만?)
  - "정부 규제 때문에" (개인정보보호법)
  - "블라인드에서 욕먹어서" (평판 문제)

- **Product Trio 확장:**
  - **PM:** 문제의 비즈니스 영향, 우선순위
  - **엔지니어:** 기술적 원인 (성능, 복잡도, 버그)
  - **디자이너:** UX 관점의 원인 (혼란, 복잡함, 시각적 이해 부족)


### ⚠️ 단계별 예외 처리

| 단계 | 예외 상황 | 대처법 |
|------|-----------|--------|
| 1단계 | 증상이 너무 많음 | 가장 빈번하거나 영향이 큰 증상 3개만 선택하세요 |
| 2단계 | 원인을 모르겠음 | "왜?"를 3번 반복하세요 (5 Whys 기법). AI에게 "이 증상의 근본 원인을 3단계로 분석하세요" |
| 3단계 | 검증 데이터 부족 | 정량 데이터가 없으면 정성 데이터(인터뷰, 피드백)로 대체 가능 |
| 4단계 | 문제 정의가 너무 넓음 | "누구의", "어떤 상황에서" 조건을 추가해서 범위를 좁히세요 |

## 🔧 Layer 3: Build & Automate

> **이 레이어는 누구를 위한 건가요?**
> AI 엔지니어, 개발자 — 문제 정의 과정을 반복 가능한 자동화 프레임워크로 만듭니다.

### 📌 시작 전에

**이 코드가 하는 일:**
Layer 1-2에서 수동으로 했던 과정을 Python 스크립트로 자동화합니다.

**왜 자동화하나?**
제품 문제 정의는 신규 기능마다 반복되는 작업입니다. 코드로 구조화하면 팀 전체가 일관된 프레임워크를 사용할 수 있습니다.

**전제조건:**
- Python 3.8+ 설치
- 필요한 패키지 설치 (코드 상단의 import 참고)
- Anthropic API 키 발급 → 환경변수 설정: `export ANTHROPIC_API_KEY=your_key`

> 📖 더 자세한 설정 방법은 [프롬프트 사용 가이드](../meta/how-to-use.md)를 참고하세요.


### Python + API 자동화

```python
import json
from datetime import datetime
from typing import Dict, List
from anthropic import Anthropic

class ProblemFramer:
    def __init__(self, product_name: str):
        self.product_name = product_name
        self.client = Anthropic()
        self.problem_analysis = {}

    def analyze_symptoms(self, metrics: Dict, customer_feedback: List[str],
                        team_observations: List[str]) -> Dict:
        """
        증상 데이터를 분석하고 문제의 핵심 찾기
        """
        prompt = f"""
당신은 문제 재정의 전문가입니다.

{self.product_name}의 다음 증상 데이터를 분석하고,
"우리가 풀어야 할 진짜 문제"를 찾으세요.

메트릭:
{json.dumps(metrics, ensure_ascii=False, indent=2)}

고객 피드백:
{chr(10).join(customer_feedback)}

팀 관찰:
{chr(10).join(team_observations)}

다음을 포함한 분석을 제시하세요:

1. **증상 (Symptom)**: 관찰 가능한 데이터
2. **문제 (Problem)**: 증상의 원인 (3단계 깊이로)
3. **범위 (Scope)**: Who / When / Where / How Often
4. **영향 (Impact)**: 정량화된 비즈니스 영향
5. **가능한 해결책**: TOP 3 및 각각의 효과도
6. **최종 문제 명제**: 명확한 Problem Statement

JSON 형식으로 응답하세요.
        """

        message = self.client.messages.create(
            model="claude-opus-4-6",
            max_tokens=2000,
            messages=[
                {"role": "user", "content": prompt}
            ]
        )

        try:
            response_text = message.content[0].text
            import re
            json_match = re.search(r'\{.*\}', response_text, re.DOTALL)
            if json_match:
                return json.loads(json_match.group())
        except Exception as e:
            print(f"분석 오류: {e}")

        return {}

    def prioritize_problems(self, problems: List[Dict]) -> List[Dict]:
        """
        여러 문제들을 우선순위로 정렬
        """
        prompt = f"""
다음은 {self.product_name}에서 발견된 문제들입니다:

{json.dumps(problems, ensure_ascii=False, indent=2)}

각 문제를 다음 기준으로 우선순위를 매기세요:

1. **영향도** (몇 명의 고객에게, 얼마나 심한가?)
2. **긴급성** (지금 풀어야 하나, 아니면 나중도 괜찮나?)
3. **해결 난이도** (쉬운가, 어려운가?)

우선순위 점수 = (영향도 × 10) + (긴급성 × 5) - (난이도 × 2)

우선순위 정렬 결과를 JSON으로 제시하세요.
        """

        message = self.client.messages.create(
            model="claude-opus-4-6",
            max_tokens=1500,
            messages=[
                {"role": "user", "content": prompt}
            ]
        )

        return message.content[0].text

    def generate_problem_statement(self, problem_analysis: Dict) -> str:
        """
        최종 문제 명제 생성
        """
        prompt = f"""
다음 문제 분석에 기반하여, 명확한 "Problem Statement"를 작성하세요:

{json.dumps(problem_analysis, ensure_ascii=False, indent=2)}

형식:
---
## 문제 명제

[대상 사용자]는 [상황]에서
[문제/고통]을 겪고 있습니다.

현재 [현재 방식]으로 대응 중이지만,
[왜 부족한가]에서 실패합니다.

우리는 [우리 솔루션]으로 [구체적 결과]를 달성하려 합니다.

**성공 지표**: [측정 가능한 숫자]

---

또한 다음도 포함하세요:
- 문제의 영향도 (명량화)
- 우선순위 평가 (HIGH/MEDIUM/LOW)
- 다음 단계 (이 문제를 풀기 위한 구체적 액션)
        """

        message = self.client.messages.create(
            model="claude-opus-4-6",
            max_tokens=1500,
            messages=[
                {"role": "user", "content": prompt}
            ]
        )

        return message.content[0].text

    def to_markdown(self, analysis: Dict, statement: str) -> str:
        """마크다운으로 내보내기"""
        output = f"# {self.product_name} 문제 재정의 분석\n\n"
        output += f"분석일시: {datetime.now().isoformat()}\n\n"

        output += "## 분석 결과\n\n"
        output += json.dumps(analysis, ensure_ascii=False, indent=2)

        output += f"\n\n## 최종 문제 명제\n\n{statement}"

        return output

# 사용 예
if __name__ == "__main__":
    framer = ProblemFramer("Notion")

    # 증상 데이터
    metrics = {
        'mau': 5000,
        'dau': 1500,
        'retention_7day': 0.45,
        'retention_30day': 0.25,
        'churn_rate': 0.10
    }

    customer_feedback = [
        "사용자 A: 가입해서 설정하는데 너무 복잡해서 포기했어요",
        "사용자 B: 속도가 느려서 일단 대기 중",
        "사용자 C: 팀원들이 안 쓰니까 나도 안 써"
    ]

    team_observations = [
        "CS: 가입 문제 관련 문의 월 50건",
        "데이터: 온보딩 플로우에서 70% 이탈",
        "영업: 데모 시 고객이 '설정이 어디?'라고 자주 물음"
    ]

    # 분석 실행
    analysis = framer.analyze_symptoms(metrics, customer_feedback, team_observations)
    statement = framer.generate_problem_statement(analysis)

    # 마크다운으로 내보내기
    with open("problem-analysis-notion.md", "w", encoding="utf-8") as f:
        f.write(framer.to_markdown(analysis, statement))

    print("문제 재정의 분석이 완료되었습니다!")
```

```
# Notion 문제 재정의 분석

## 1단계: 증상 (Symptom)

**관찰된 데이터**:
- 신규 사용자 7일 리텐션: 45% (업계 평균: 60%)
- 가입 완료자 중 첫 프로젝트 생성 비율: 30% (목표: 80%)
- 고객 지원 문의 TOP 3: "설정이 어디에?", "이건 뭐 하는 기능?", "너무 복잡"
- 스크린 레코딩: 신규 사용자 평균 온보딩 시간 20분

## 2단계: 문제 (Problem) — 3단계 깊이

**Level 1: 넓은 문제**
→ "신규 사용자 온보딩이 효과적이지 않다"

**Level 2: 구체화**
→ "초기 설정 프로세스가 길어서, 신규 사용자가 포기하기 전에 가치를 못 느낀다"

**Level 3: 근본 원인**
1. **프로세스의 길이**: 이메일 인증 → 이름 입력 → 첫 Workspace 생성 → 초대 설정 = 총 4단계
2. **인지적 부담**: "Database란?", "Template란?" 등 새로운 개념 설명
3. **즉시 보상 부족**: "이게 뭐 좋은 거지?" 느낄 시간 없음
4. **포기 포인트**: 3번째 단계 (초대 설정)에서 60% 이탈

고객 인터뷰:
- "좋아 보이는데, 설정이 많아서 나중에 하려다가 잊어버렸어요"
- "뭔가 시작해 보고 싶은데, 어디서부터 시작해야 할지 모르겠어요"
- "팀 초대 설정이 필요한데, 일단 혼자 시작하고 싶었어요"

## 3단계: 범위 (Scope)

**Who**: 신규 가입 사용자 전체 (특히 회사 규모 1-10명 스타트업)
**When**: 첫 48시간 (가입 직후)
**Where**: 온보딩 플로우 (가입 후 첫 프로젝트까지)
**How Often**: 신규 사용자 100% (매우 일반적)

→ 범위: 월 신규 가입 500명 중 55% (275명) 이탈

## 4단계: 영향 (Impact) — 정량화

**규모**:
- 월 신규 가입: 500명
- 7일 이탈: 275명 (55%)
- 각 사용자 LTV: 평균 $80 (연 구독료)

**재정적 영향**:
- 월 손실: 275명 × $80 ÷ 12 = 월 $1,833 손실

**비즈니스 영향**:
- "만약 리텐션을 45% → 70%로 개선하면?"
- 월 신규 500명 × 30% 개선 = 월 150명 추가 확보
- 월 추가 수익: 150명 × $80 ÷ 12 = 월 $1,000 증가
- 연간: +$12,000 수익 증가

**심각성**: HIGH (고객이 가치를 못 느낀 채로 떠남)
**빈도**: VERY HIGH (모든 신규 사용자)

**우선순위 점수**: (275 × 10) + (심각도 10 × 5) - (해결 난이도 6 × 2) = 2,858 (매우 높음)

## 5단계: 해결책 비교

**옵션 1**: 온보딩 플로우 단순화 (4단계 → 2단계)
- 혜택: 시간 20분 → 10분 (50% 개선)
- 리텐션 개선: 45% → 55% (10%p)
- 비용: 낮음 (UI 개선만)
- 추정 ROI: 월 +$500

**옵션 2**: "스킵" 버튼 추가 (필수 설정만 남김)
- 혜택: 시간 20분 → 3분 (85% 개선)
- 리텐션 개선: 45% → 70% (25%p)
- 비용: 중간 (기능 개발)
- 추정 ROI: 월 +$1,000

**옵션 3**: 인터랙티브 튜토리얼 (가이드)
- 혜택: "뭐 하는 거지?" 해소
- 리텐션 개선: 45% → 60% (15%p)
- 비용: 중간
- 추정 ROI: 월 +$750

→ **최고 효율**: 옵션 2 (비용 대비 ROI 최고)

## 6단계: 최종 문제 명제

---
### 문제 명제

**신규 사용자 (특히 1-10명 스타트업 PM)**은 Notion을 처음 가입하면
**복잡한 초기 설정 프로세스 (20분, 4단계)**를 겪고 있습니다.

현재 모든 설정을 필수로 진행하도록 설계했지만,
**신규 사용자가 설정을 완료하기 전에 제품 가치를 못 느껴서 55%가 포기**합니다.

우리는 **"스킵" 버튼으로 즉시 시작 가능하게 (3분 안에)**하고,
**필수 설정은 나중에 할 수 있도록** 재설계하여
**신규 사용자 7일 리텐션을 45% → 70%로 개선**하려 합니다.

**성공 지표**:
- 온보딩 완료 시간: 20분 → 3분
- 첫 프로젝트 생성 비율: 30% → 80%
- 7일 리텐션: 45% → 70%
- 월 신규 추가 확보: +150명

---

## 다음 단계

1. 팀 동의 (PM, 엔지니어, 디자이너가 이 문제에 동의하나?)
2. 솔루션 설계 (옵션 2를 어떻게 구현할지)
3. 실험 설계 (A/B 테스트로 효과 검증)
4. 실행 및 모니터링
```
