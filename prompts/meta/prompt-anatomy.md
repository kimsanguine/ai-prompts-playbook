# 🔬 프롬프트 해부학 (Prompt Anatomy: Reverse Engineering)

> 예상 소요: 10분 (Quick) / 25분 (Deep) / 45분+ (Build)
> 워크플로우: ← `prompt-builder.md`를 이해한 후

---

## 🎯 이 프롬프트가 해결하는 문제

좋은 프롬프트를 분석하면 **"왜 좋은가?"를 역공학할 수 있다.** 이 프롬프트는 좋은 프롬프트를 분해(Decompose)해서 각 요소를 파악하고, 그 구조를 다른 프롬프트에 적용할 수 있게 가르친다. **"좋은 프롬프트의 해부학"을 배우는 것이다.**

---

## ⚡ Layer 1: Quick Start

> **이 레이어는 누구를 위한 건가요?**
> AI를 처음 쓰는 분, 비개발자, 학생 — 5분이면 좋은 프롬프트의 구조를 이해할 수 있습니다.

### 📌 사용법

1. **ChatGPT, Claude, Gemini 중 아무 AI 챗봇**을 열고 새 대화를 시작하세요
2. 아래 프롬프트를 **통째로 복사**해서 AI에 붙여넣으세요
3. `{중괄호}` 안의 내용만 여러분의 상황에 맞게 바꾸세요
4. Enter를 누르면 끝!

> 📖 더 자세한 사용법은 [프롬프트 사용 가이드](../meta/how-to-use.md)를 참고하세요.


```
당신은 프롬프트 분석 전문가다.

"좋은 프롬프트"를 분해하세요:

1. 역할 (Role): 어떤 관점에서 답하는가?
2. 맥락 (Context): 배경 정보는?
3. 질문 (Core Question): 핵심 질문은?
4. 제약 (Constraints): 한계/조건은?
5. 추론 (Reasoning): "어떻게 생각하세요?"라는 지시가 있나?
6. 형식 (Format): 답변 형식은?
7. 예시 (Examples): 구체 예시가 있나?
8. 톤 (Tone): 어떤 어조인가?

각 요소별로:
- 무엇인가?
- 왜 효과적인가?
- 다른 프롬프트에 어떻게 적용?

분석해주세요.
```

💡 **Quick Tips:**
- 좋은 프롬프트는 "구조"가 있다 → 분석하면 패턴 보임
- 패턴을 알면 → 나만의 프롬프트도 좋아짐
- 예: "역할 + 구체 질문 + 제약 + 형식" = 80% 성공


### ⚠️ 이런 경우는요?

| 상황 | 대처법 |
|------|--------|
| 프롬프트 엔지니어링이 처음 | 이 문서를 먼저 읽고, 그 다음 prompt-builder.md로 실습하세요 |
| 분석할 프롬프트가 없음 | 이 Playbook의 프롬프트를 대상으로 연습하세요 |
| 결과가 좋은 프롬프트를 왜 분석하나 | "왜 좋은지" 이해하면 다른 프롬프트에도 적용 가능 |
| 영어 프롬프트와 한국어의 차이 | 역할/목표/맥락 구조는 동일. 표현만 언어에 맞게 조정 |

## 🧠 Layer 2: Deep Dive

> **이 레이어는 누구를 위한 건가요?**
> PM, 기획자, 마케터 — 프롬프트 엔지니어링의 원리를 깊이 이해하고 싶은 분입니다.
"왜 이 프롬프트가 잘 작동하는가?"를 역공학으로 분석하고 싶다면 이걸 읽어보세요.

### 📌 진행 방식

이 프롬프트는 **5단계 CoT(Chain of Thought)** 구조입니다. 두 가지 방식으로 쓸 수 있습니다:

**방식 A: 한 번에 넣기 (추천)**
- 아래 전체를 통째로 복사해서 AI에 붙여넣으세요
- AI가 모든 단계를 순서대로 실행합니다
- 결과가 한 번에 나옵니다 (길지만 완결적)

**방식 B: 단계별로 넣기 (더 깊은 분석)**
- 1-2단계를 먼저 넣고 결과를 확인하세요
- 결과가 만족스러우면 다음 단계를 이어서 넣으세요
- 중간에 수정이 필요하면 해당 단계를 수정 요청할 수 있습니다
- 시간이 더 걸리지만, 각 단계를 직접 검증할 수 있습니다

**`<!-- 💬 -->` 교육 주석이란?**
- 프롬프트 안에 `<!-- 💬 ... -->` 형태의 주석이 있습니다
- 이건 **여러분을 위한 설명**이지, AI에게 보내는 지시가 아닙니다
- AI에 붙여넣을 때 포함해도 되고 빼도 됩니다 (결과에 영향 없음)
- "왜 이렇게 설계했는가?"를 이해하면 더 좋은 프롬프트를 만들 수 있습니다

> 📖 더 자세한 사용법은 [프롬프트 사용 가이드](../meta/how-to-use.md)를 참고하세요.


### 역할 설정

당신은 **프롬프트 구조 분석가(Prompt Architecture Analyst)**이다. 당신의 역할:

1. **구조 파악**: 좋은 프롬프트의 "뼈대" 식별
2. **요소 분해**: 각 요소가 "무엇을 하는가?" 이해
3. **재적용**: 패턴을 새로운 상황에 적용

### 목표

최종 아웃풋:
- **프롬프트 해부 도표** (8개 요소와 상호작용)
- **패턴 분석** ("왜 이 조합이 효과적인가?")
- **체크리스트** (나만의 프롬프트 검증용)
- **실습 사례** (분석 예시 3-5개)

### Reasoning Strategy

**Step 1: 프롬프트 해부학의 8가지 요소 이해**

```
좋은 프롬프트의 구성 요소:

1️⃣  ROLE (역할)
   문제: "누가 이 질문에 답하는가?"
   역할 없음: "요약해줄 수 있어?"
   역할 있음: "당신은 법무팀장입니다. 요약해주세요."
   효과: 맥락 제공, "관점" 설정

2️⃣  CONTEXT (맥락)
   문제: "왜? 어떤 배경?"
   맥락 없음: "메트릭을 정의해줄 수 있어?"
   맥락 있음: "AI 출시 후 3개월, 채택률이 15%에 미쳤어.
               이유를 파악하기 위해 메트릭을 재설계하고 싶은데..."
   효과: 문제의 우절감 전달, 긴급도 부여

3️⃣  CORE QUESTION (핵심 질문)
   문제: "정확히 뭘 물어볼 건가?"
   약한 질문: "좋은 아이디어 있어?"
   강한 질문: "신뢰도가 낮은 사용자(4/10 이하)를 위해,
              신뢰도를 올리는 3가지 방안을 제시해주세요."
   효과: 답변의 초점 명확화

4️⃣  CONSTRAINTS (제약)
   문제: "어떤 한계/조건이 있나?"
   제약 없음: 답변이 산으로 감
   제약 있음: "한국 기업 관점만. 비용은 $10k 이하.
              1주일 안에 구현 가능한 것만."
   효과: 답변의 범위 한정, 실행 가능성

5️⃣  REASONING (추론 지시)
   문제: "어떻게 생각하라는 건가?"
   추론 지시 없음: "이게 될까?" → "예/아니오"
   추론 지시 있음: "다음 3단계로 분석해주세요:
                  1) 기술 가능성
                  2) 비용 효율성
                  3) 사용자 영향도
                  각 단계별로 설명 후 결론"
   효과: 답변의 "생각 과정" 체계화

6️⃣  FORMAT (형식)
   문제: "답변을 어떤 형식으로 원하나?"
   형식 없음: "설명해줘" → 자유 형식
   형식 있음: "마크다운 표로.
              | 방안 | 장점 | 단점 | 예상비용 |"
   효과: 답변의 구조화, 재사용 용이성

7️⃣  EXAMPLES (예시)
   문제: "구체적인 예는?"
   예시 없음: "요약해줄 수 있어?"
   예시 있음: "예를 들어,
              Input: '이 계약서의 위험은?'
              Output: '위험 조항 3개 + 신뢰도 7.2/10
                       + 권장 액션'"
   효과: 패턴 전달, 정확도 3배 ↑

8️⃣  TONE (톤)
   문제: "어떤 어조로?"
   톤 없음: 모호함
   톤 있음: "전문적이지만 초보자도 이해할 수 있게.
            기술 용어는 최소화."
   효과: 답변의 "느낌" 제어
```

**Step 2: 8가지 요소의 상호작용**

```
프롬프트의 "화학" (상승작용):

ROLE + CONTEXT
→ "누가 어떤 상황에서" = 몰입도 ↑

CORE QUESTION + REASONING
→ "뭘 물어볼 건가?" + "어떻게 생각하라?" = 정확도 ↑

CONSTRAINTS + FORMAT
→ "어떤 한계?" + "어떤 형식?" = 실행 가능성 ↑

EXAMPLES + TONE
→ "구체 사례" + "올바른 톤" = 신뢰도 ↑

모두 합치기:
ROLE + CONTEXT → CORE QUESTION
+ REASONING → FORMAT
+ EXAMPLES + TONE + CONSTRAINTS
= "완벽한 프롬프트"
```

**Step 3: 실전 분석 예시**

```markdown
## 사례: "AI 채택 메트릭 정의" 프롬프트 분해

원본 프롬프트:
"당신은 AI 제품 메트릭 전문가입니다. 15년 경력.
우리는 AI 요약 기능을 3개월 전 출시했는데,
채택률이 15%로 목표(25%)를 못 미쳤습니다.
이유를 파악하고 신뢰도를 높이기 위해,
다음 기준으로 메트릭을 재설계해주세요:

1) 사용자가 이해할 수 있어야 함
2) SQL로 자동 계산 가능
3) 3-5개만 (너무 많으면 관리 어려움)
4) 한국 기업 관례 반영

출력: 마크다운 표
| 메트릭 | 정의 | 측정 | 목표 | 해석 |

예시:
Activation Rate: 가입자 중 첫 사용 비율 = ...
Trust Score: 신뢰도 점수 = ..."

분석:

✅ ROLE (8/10)
- "AI 제품 메트릭 전문가" ✓
- "15년 경력" ✓ (구체적 경험 명시)
- 약점: 성향 (데이터 기반? 실용 지향?) 명시 안 함

✅ CONTEXT (9/10)
- 상황 설명 (출시 후 3개월) ✓
- 문제 명시 (15% vs 25% 목표) ✓
- 배경 (왜 다시 설계? = 신뢰도 향상 필요) ✓

✅ CORE QUESTION (8/10)
- 분명한 요청 (메트릭 재설계) ✓
- 약점: "왜 15%인가?"에 대한 가설 분석 요청 없음

✅ CONSTRAINTS (9/10)
- 명확한 4가지 제약 ✓
- "3-5개만" (범위 제한) ✓
- "한국 기업 관례" (문화적 맥락) ✓

✅ REASONING (7/10)
- 암묵적 (메트릭 설계의 단계는 명시 안 함)
- 개선: "먼저 채택 저조의 원인 파악 → 메트릭 정의" 명시

✅ FORMAT (10/10)
- 마크다운 표 형식 명확 ✓
- 헤더 명시 ✓

✅ EXAMPLES (8/10)
- 1개 예시 있음 ✓
- 개선: "Trust Score" 예시도 추가하면 완벽

✅ TONE (7/10)
- 암묵적 (전문적 톤)
- 개선: "초보자도 이해할 수 있게" 명시

전체 점수: 8.3/10 (매우 좋은 프롬프트)

개선 가능성:
- REASONING 명확화: "Step 1: 저조 원인 분석 → Step 2: 메트릭 설계"
- TONE 명시: "전문 용어 최소화, 누구나 이해 가능"
- 추가 예시: Trust Score 정의 예시
```

**Step 4: 패턴 추출**

```
공통 패턴 (모든 좋은 프롬프트에서):

패턴 1: Role + Context = "깊은 이해"
→ 모든 성공한 프롬프트는 상황을 먼저 설정함

패턴 2: Question + Reasoning = "체계적 사고"
→ 질문 + "어떻게 생각할 건가?"의 조합

패턴 3: Constraints + Format = "실행 가능"
→ 한계를 명확히하면 결과를 구조화 가능

패턴 4: Examples + Tone = "신뢰"
→ 구체 사례 + 적절한 톤 = 사용자 신뢰

→ 이 4가지 패턴을 조합하면 80% 성공
```

### 프롬프트 해부 체크리스트

좋은 프롬프트를 만들 때:

```
□ ROLE: 누가 답할 건가? (직책 + 경험 + 성향)
  [] Role이 명확한가?
  [] 배경/경험이 구체적인가?
  [] 관점/성향이 명시되어 있는가?

□ CONTEXT: 왜? 어떤 배경?
  [] 현재 상황 설명?
  [] 문제점 명시?
  [] 긴급도 느껴지는가?

□ CORE QUESTION: 정확히 뭘 물어볼 건가?
  [] 분명한가?
  [] 답변이 한 방향으로 유도되는가?
  [] 구체적인가?

□ REASONING: 어떻게 생각하라는 건가?
  [] 단계적 분석 지시?
  [] 사고 프레임워크 제시?

□ CONSTRAINTS: 어떤 한계?
  [] 범위 제한?
  [] 길이/형식 제한?
  [] 문화적/산업적 맥락?

□ FORMAT: 답변 형식?
  [] 마크다운? JSON? 자유 형식?
  [] 헤더 명시?
  [] 테이블? 리스트?

□ EXAMPLES: 구체 예는?
  [] 1개 이상?
  [] Input/Output 쌍?
  [] 패턴이 명확한가?

□ TONE: 어떤 어조?
  [] 전문적? 캐주얼?
  [] 기술 용어 수준?
  [] 대상 청중 맞는가?

점수: 각 항목 ✓ = 1점
7-8점: 우수한 프롬프트
5-6점: 개선 필요
0-4점: 근본 개편 필요
```

💡 **Deep Tips:**
- **ROLE이 가장 중요**: 역할 정의 하나로 절반 이상의 정확도 결정
- **EXAMPLES의 힘**: 예시 1개 = 1,000단어 설명 같은 효과
- **CONSTRAINTS는 "선물"**: 범위를 좁히면 답변 품질이 올라감


### ⚠️ 단계별 예외 처리

| 단계 | 예외 상황 | 대처법 |
|------|-----------|--------|
| 1단계 | 구성 요소 파악 어려움 | "역할, 목표, 맥락, 지시, 제약"으로 분류해보세요 |
| 2단계 | 효과적 패턴 식별 어려움 | "이 프롬프트에서 빼면 결과가 나빠지는 부분"을 찾으세요 |
| 3단계 | 비교 분석 어려움 | 같은 목표의 프롬프트 2개를 비교해서 차이를 분석 |
| 4단계 | 개선 포인트 모름 | "결과에서 가장 아쉬운 부분"부터 역추적 |

## 🔧 Layer 3: Build & Automate

> **이 레이어는 누구를 위한 건가요?**
> AI 엔지니어, 개발자 — 프롬프트 품질 분석과 개선 제안을 자동화합니다.

### 📌 시작 전에

**이 코드가 하는 일:**
Layer 1-2에서 수동으로 했던 과정을 Python 스크립트로 자동화합니다.

**왜 자동화하나?**
프롬프트를 체계적으로 분석하면, 어떤 요소가 결과 품질에 영향을 주는지 패턴을 발견할 수 있습니다.

**전제조건:**

### Claude Code CLI

```bash
# 1. 프롬프트 해부
claude code run analyze-prompt-anatomy \
  --prompt-file my_prompt.txt \
  --output anatomy_report.md

# 2. 여러 프롬프트 비교
claude code run compare-prompts \
  --prompt-1 good_prompt.txt \
  --prompt-2 bad_prompt.txt \
  --aspect role,context,format

# 3. 패턴 추출
claude code run extract-patterns \
  --prompt-collection successful_prompts/ \
  --output pattern_analysis.json

# 4. 프롬프트 체크리스트 생성
claude code run generate-checklist \
  --prompt-file my_prompt.txt \
  --format pdf
```
