# 🎨 AI Native PM 설계 원칙 (Design Principles for AI Native PMs)

> 이 문서는 "프롬프트"가 아니라 **"설계 철학 가이드"**입니다.
> AI PM 업무의 모든 프롬프트를 통해 관통하는 7가지 원칙을 설명합니다.

---

## 개요

AI 제품(AI를 핵심 기능으로 하는 제품)을 기획할 때, 일반 SaaS PM과는 다른 사고방식이 필요하다. **AI Native PM의 7가지 설계 원칙**은 이 문서에서 제시하는 모든 프롬프트의 기초가 되는 철학이다.

이 원칙들을 따르면:
- AI 기능의 복잡성을 관리할 수 있다
- 팀 간 오해와 갈등을 줄인다
- 사용자 신뢰를 확보한다
- 출시 후 문제를 조기에 발견한다

---

## 원칙 1️⃣: 질문과 답변 구조 (Question-Answer Architecture)

### 정의

AI는 "답변을 생성하는 시스템"이다. AI 기능을 기획할 때 항상 먼저 **"사용자가 어떤 질문을 하고 있는가?"**를 명확히 해야 한다.

### 원칙

```
❌ 잘못된 접근:
"AI가 자동으로 요약해주는 기능"
→ 기술 중심, 목표 불명확

✅ 올바른 접근:
"사용자의 질문: '이 100페이지 계약서의 핵심이 뭐야?'
 AI의 답변: '핵심 5줄 + 신뢰도 점수'"
→ 사용자 중심, 기능 명확
```

### 프레임워크

```
Step 1: 사용자 질문 정의
- 질문: "이 문서의 위험 조항은 뭔가?"
- 맥락: 법무팀이 계약 검토 중
- 시간 압박: "5분 안에 판단해야 해"
- 기대치: "완벽하진 않아도 80% 정도면 OK"

Step 2: AI의 답변 형식 정의
- 형식: "위험 조항 3개 + 신뢰도"
- 세부: 조항 이유 + 권장 액션
- 신뢰도: 낮으면 "전문가 검토 필수" 경고

Step 3: 피드백 루프 정의
- 사용자가 "틀렸네"라고 지적 → AI가 학습
- 또는 "맞아"라고 확인 → 신뢰도 상승
- → 모델 성능 개선 → 다음 사용자를 위한 더 나은 답변
```

### 예시 (금융 제품에서)

```
질문: "이 투자 제안서의 리스크는?"
↓
AI 분석: "기술 리스크 3개, 재무 리스크 2개 detected"
↓
신뢰도: 7.8/10 (높음)
↓
사용자: "맞네, 내 예상과 같다" (확인)
↓
모델: 이 패턴을 학습 → 다음 요청에 더 정확
```

### 이 원칙이 다른 프롬프트에 반영되는 방식

- **measure/ai-adoption-metrics.md**: "신뢰도"는 "사용자가 AI의 답변을 믿는가?"를 측정
- **communicate/storytelling.md**: PM의 스토리는 "사용자가 어떤 질문을 하고 있었나?"로 시작
- **meta/prompt-builder.md**: "좋은 프롬프트"는 "명확한 질문"을 포함

## 원칙 2️⃣: 데이터 기반 가설 (Data-Driven Hypotheses)

### 정의

AI 기능의 모든 의사결정은 **"데이터 기반 가설"**이어야 한다. 직감이나 "좋은 아이디어"가 아니라, 실제 사용자 데이터/피드백으로 검증된 가설만 채택한다.

### 원칙

```
❌ 잘못된 사고:
"요약 기능이 좋을 것 같은데, 만들어보자"
→ 낭비, 실패 위험

✅ 올바른 사고:
"고객 100명 중 85명이 '문서 읽기에 시간 낭비'라고 답했다"
→ "데이터 기반 가설: AI 요약으로 80% 시간 단축 가능"
→ 출시 후: 실제 달성률 (예: 75% 달성) 측정
→ 피드백 반영: "신뢰도 표시 추가" 개선
```

### 프레임워크

```
가설의 3가지 요소:

1. 데이터 (Data)
   "고객 조사: 법무팀 20명 중 18명이 '계약서 검토 시간 최소화 원함'"

2. 예상 효과 (Expected Impact)
   "AI 요약으로 검토 시간 2시간 → 20분으로 단축 (90% 절감)"

3. 검증 방법 (Validation Method)
   "Beta 사용자 20명에게 2주간 사용 후 ' 실제 시간 단축'은 얼마나? 측정"

→ 결과: 60% 단축 (90% 예상 vs 60% 실제)
→ 해석: "신뢰도가 낮아서 여전히 검증하는 데 시간 씀"
→ 개선: "신뢰도 표시 추가"
```

### 이 원칙이 다른 프롬프트에 반영되는 방식

- **measure/**: 모든 메트릭은 "가설 검증"을 위한 것
- **measure/ab-test-design.md**: A/B 테스트로 "어느 가설이 맞는가?" 검증
- **communicate/stakeholder-alignment.md**: 경영진 설득 자료는 "데이터 + 가설"

## 원칙 3️⃣: 추론 전략 (Reasoning Strategy)

### 정의

AI 모델의 성능을 높이려면, 모델이 "어떻게 생각하는가?"를 이해해야 한다. 이를 **Reasoning Strategy**라 한다: "AI가 결론에 도달하기까지의 생각 과정"을 명시하고 최적화한다.

### 원칙

```
❌ 나쁜 프롬프트 (Reasoning 없음):
"이 계약서를 요약하세요"
→ 모델이 임의로 중요도 판단
→ 결과가 일관성 없음

✅ 좋은 프롬프트 (Reasoning 있음):
"이 계약서를 다음 순서로 분석:
 1. 당사자 식별
 2. 주요 의무사항
 3. 위험 조항
 4. 금액/기간
 5. 종료 조건
그 다음 '법무팀이 가장 관심 있을 3가지' 요약"
→ 일관성 있는 결과
→ 신뢰도도 올라감
```

### 프레임워크

```
Reasoning Strategy 설계:

Step 1: 목표 분해
질문: "이 보고서의 리스크는?"
↓
분해:
1. 재무 리스크는?
2. 기술 리스크는?
3. 시장 리스크는?
4. 규제 리스크는?

Step 2: 각 단계별 판단 기준 제시
재무 리스크:
- 수익성 < 30% → 높은 리스크
- 부채비율 > 70% → 높은 리스크
- 현금 보유 < 3개월 → 높은 리스크

Step 3: 신뢰도 계산
"위 4개 리스크 중 몇 개가 발견되었는가?"
- 0-1개: 신뢰도 9/10 (안전)
- 2-3개: 신뢰도 6/10 (주의)
- 4개 이상: 신뢰도 3/10 (위험)
```

### 한국 기업 문화에 맞는 Reasoning

```
한국식 의사결정은 "단계적 검증"을 선호:

프롬프트에 명시:
"이 제안서를 다음 5단계로 검토:
 1. 기본 사항 확인 (주체, 금액, 기간)
 2. 팀장급 리뷰 포인트 (전략성, ROI)
 3. 임원급 리뷰 포인트 (리스크, 규제)
 4. 최종 의사결정 기준
 5. 각 단계별 신뢰도"

→ AI가 "각 의사결정 단계"를 명확히 함
→ 한국식 회사의 "결재 프로세스"와 정렬
→ 신뢰도 향상
```

### 이 원칙이 다른 프롬프트에 반영되는 방식

- **measure/ab-test-design.md**: A/B 테스트 설계할 때 "모델의 reasoning이 달라지는가?"를 확인
- **communicate/storytelling.md**: "고객의 생각 과정 변화"를 스토리로 표현
- **meta/prompt-builder.md**: "좋은 프롬프트"의 필수 요소는 "명시적 reasoning step"

## 원칙 4️⃣: 워크플로우 체인 (Workflow Chain)

### 정의

AI 기능은 **"일회성 답변"이 아니라 "반복적 워크플로우"의 일부**여야 한다. 한 번의 AI 결과 → 사용자 검증 → 피드백 → 액션 → 다음 단계로의 체인을 설계한다.

### 원칙

```
❌ 잘못된 설계 (일회성):
사용자: "요약해줘"
AI: "요약 결과 표시"
사용자: "끝"
→ 피드백 없음, 개선 불가

✅ 올바른 설계 (워크플로우 체인):
사용자: "요약해줘"
↓
AI: "요약 + 신뢰도 표시"
↓
사용자: "신뢰도가 낮네? [직접 검토] / 신뢰도 높네? [바로 사용]"
↓
(신뢰도 낮은 경우) 사용자 수정 입력
↓
AI: 수정사항 학습
↓
다음 유사 문서 → AI 신뢰도 향상
```

### 프레임워크

```
Workflow Chain 설계:

Phase 1: AI 생성 (Generative)
→ AI가 초안/요약 생성
→ 신뢰도 점수 표시

Phase 2: 사용자 검증 (Verification)
→ 사용자가 "맞음/틀림" 피드백
→ 또는 직접 수정

Phase 3: 피드백 수집 (Feedback)
→ "어떤 부분이 틀렸나?"
→ "왜 이렇게 생각했나?"

Phase 4: 모델 개선 (Improvement)
→ 피드백을 학습 데이터로 변환
→ 모델 재학습
→ 신뢰도 상승

Phase 5: 다음 사용자 혜택 (Amplification)
→ 다음 사용자가 더 정확한 AI 결과 받음
→ 전체 시스템의 신뢰도 상승
```

### 구체 예시 (법무팀 계약서 검토)

```
Day 1:
팀원A: "AI 요약으로 계약서 검토"
→ 신뢰도 6.5/10 (중간) → 직접 검토
→ "이 조항 놓쳤네" 피드백

Day 2-7:
"유사한 계약서 100개"에 대해 AI 결과 + 피드백 축적

Day 8:
팀원B: 같은 유형의 새 계약서
→ AI가 이전 피드백 학습
→ 신뢰도 8.2/10 (높음) → 바로 사용
→ 시간 단축 (60분 → 10분)

시간이 지날수록:
AI 신뢰도 ↑ → 검증 시간 ↓ → 효율성 ↑
```

### 이 원칙이 다른 프롬프트에 반영되는 방식

- **measure/ai-adoption-metrics.md**: "신뢰도 → 채택 → 리텐션"의 체인을 추적
- **measure/retention-analysis.md**: 워크플로우 체인이 "반복 사용"으로 이어지는지 측정
- **communicate/storytelling.md**: "변화의 여정"은 사실 "워크플로우 체인의 개인 버전"

## 원칙 5️⃣: 자기 검증 (Self-Verification)

### 정의

AI의 신뢰도를 높이려면, **AI 스스로 "내 답변의 불확실성"을 평가**해야 한다. 이를 **Self-Verification**: "AI가 자신의 답변 신뢰도를 스스로 점수화"한다.

### 원칙

```
❌ 잘못된 설계:
AI: "이 계약서는 위험합니다" (신뢰도 없음)
→ 사용자: "이 AI를 얼마나 믿어야 되?"
→ 신뢰도 저하

✅ 올바른 설계:
AI: "이 계약서는 위험합니다 (신뢰도 8.2/10)"
설명: "법적 선례 분석에서는 높은 신뢰도이나,
       금융 조항 부분은 데이터 부족으로 신뢰도 낮음"
→ 사용자: "아, 법무팀은 꼭 검토하되, 재무팀 의견도 들어야겠네"
→ 신뢰도 향상 + 더 나은 의사결정
```

### 프레임워크

```
Self-Verification 설계:

Step 1: 신뢰도 점수 계산
"내(AI)가 이 답변을 얼마나 확신하는가?"

메커니즘:
- 학습 데이터 확실성: "이 유형의 데이터가 충분한가?" (0-100%)
- 모델 신뢰도: "이 분야에서 과거 정확도는?" (0-100%)
- 입력 명확도: "사용자의 질문이 명확한가?" (0-100%)

신뢰도 = (학습데이터 신뢰도 × 0.4) + (모델신뢰도 × 0.4) + (입력명확도 × 0.2)

Step 2: 신뢰도별 권장사항 제시
신뢰도 8-10: "이 결과를 신뢰하고 바로 액션 가능"
신뢰도 6-7: "이 결과를 참고하되, 의심 부분은 직접 검증"
신뢰도 4-5: "전문가 검증 필수. AI는 참고용만"
신뢰도 0-3: "신뢰할 수 없음. 전문가 의견 필수"
```

### 한국 기업 문화와의 정렬

```
한국 조직: "다중 검증" (팀장→임원→최종결재) 중시

Self-Verification과의 정렬:
AI 신뢰도 8/10 → 팀장 검증만 필요
AI 신뢰도 5/10 → 팀장 + 임원 검증 필요
AI 신뢰도 2/10 → 전체 검증 프로세스 필요

→ "AI 신뢰도"가 "회사 검증 프로세스"를 자동화
→ 불필요한 단계 제거 → 의사결정 속도 ↑
```

### 이 원칙이 다른 프롬프트에 반영되는 방식

- **measure/ai-adoption-metrics.md**: "신뢰도 점수"는 Self-Verification을 측정하는 핵심 메트릭
- **communicate/storytelling.md**: 고객이 "신뢰도를 이해하는 과정"은 "일의 의미 변화" 표현
- **measure/ab-test-design.md**: A/B 테스트에서 "신뢰도 변화"는 Primary Metric

## 원칙 6️⃣: 레이어드 접근성 (Layered Accessibility)

### 정의

AI 기능을 사용하는 사람의 수준/배경이 다르다. **"쉬운 사용부터 깊은 이해까지" 단계적으로 접근 가능**하게 설계한다.

### 원칙

```
❌ 잘못된 설계 (일괄적):
모든 사용자에게 같은 수준의 UI/설명
→ 초보자는 너무 복잡하고, 전문가는 너무 단순

✅ 올바른 설계 (레이어드):

Layer 1 (Casual User, 1분):
"AI가 핵심을 정리해줍니다" (간단한 설명)
"신뢰도: 7/10" (숫자만)

Layer 2 (Regular User, 5분):
"신뢰도 7/10이라는 건 뭐죠?"
→ "법적 선례는 95% 정확도, 금융은 50% 정확도" (세부)

Layer 3 (Expert/Admin, 20분):
"모델 성능 상세 분석"
→ Confusion Matrix, Precision/Recall by category
→ 모델 재학습 옵션
```

### 프레임워크

```
Layered Design:

Interface Layer (사용자 인터페이스):
- 기본: "신뢰도 7/10"
- 클릭하면: 설명 확장
- 고급: "상세 분석 보기" 링크

Education Layer (교육/설명):
- 빠른 가이드: 1분 (기본 사용법)
- 중간 가이드: 5분 (신뢰도 해석)
- 깊은 가이드: 20분 (모델 원리)

Control Layer (제어):
- 기본: AI 결과 보기만
- 중급: 결과 편집 가능
- 고급: 모델 파라미터 조정
```

### 이 원칙이 다른 프롬프트에 반영되는 방식

- **meta/prompt-builder.md**: "이 프롬프트 자체"가 Layer 1/2/3 구조
- **communicate/storytelling.md**: "처음 경험자 → 정기 사용자 → 파워 유저" 변화
- **communicate/cross-team-handoff.md**: 각 팀에게 "필요한 수준의 정보만" 제공

## 원칙 7️⃣: 교육적 투명성 (Educational Transparency)

### 정의

AI의 의사결정 과정을 투명하게 보여줄 때, **단순히 "공개"하는 것이 아니라 "사용자가 배울 수 있도록"** 설계한다.

### 원칙

```
❌ 잘못된 투명성 (정보만 공개):
"이 신뢰도 점수는 Bayesian 확률에 기반한 Dirichlet 분포..."
→ 대부분 사용자가 이해 못 함
→ 투명성 없는 것과 같음

✅ 올바른 투명성 (배우기):
"신뢰도 7이라는 건:
 '100개 비슷한 계약서 중 70개는 우리 분석이 맞았다'는 뜻입니다.
  따라서 남은 30개 중 하나일 가능성도 있으니,
  신뢰도가 높아도 중요한 부분은 직접 검토하세요."
→ 사용자가 "신뢰도가 뭔지" 이해 + 실제 사용 시 더 나은 판단 가능
```

### 프레임워크

```
Educational Transparency 설계:

Information → Context → Action

정보 (Information):
"신뢰도: 7.2/10"

맥락 (Context):
"이게 뭘 의미하는가?"
"왜 이 정도인가?"
"이전 유사한 경우는 어땠나?"

액션 (Action):
"이 신뢰도를 받으면 어떻게 행동해야 하나?"
"신뢰도 낮으면 누구에게 물어봐야 하나?"
"더 높은 신뢰도를 원하면 뭘 해야 하나?"
```

### 구체 예시

```
❌ 기계적 투명성:
"신뢰도 7.2/10"

✅ 교육적 투명성:
"신뢰도 7.2/10 (높음)
 = 우리 경험상 이 부분 정확도가 90%

이유:
- 법무 선례 데이터: 충분함 ✅
- 한국 계약서 데이터: 충분함 ✅
- 금융 조항 데이터: 부족함 ⚠️

따라서:
- 일반 조항: 이 요약 신뢰 가능
- 금융 조항: 직접 재무팀과 검토 권장

다음 단계:
- [편집] (다른 판단 기반 추가)
- [승인] (이 요약 사용)
- [거절] (다시 분석)
→ 당신의 피드백이 우리 모델을 더 똑똑하게 만듭니다"
```

### 한국 시장에서의 의미

```
한국 사용자는 "왜?"를 중시:
- "왜 이 정도 신뢰도인가?"
- "왜 이 부분을 놓쳤나?"
- "왜 나는 다르게 생각했나?"

교육적 투명성으로 이 "왜?"에 답변:
→ AI가 "학습 기계"가 아니라 "설명 가능한 고문"이 됨
→ 신뢰도 크게 향상
```

### 이 원칙이 다른 프롬프트에 반영되는 방식

- **모든 프롬프트의 "Deep Dive" 섹션**: <!-- 💬 --> 교육 주석으로 "왜?"를 설명
- **communicate/storytelling.md**: 고객의 "깨달음"은 사실 "AI 투명성을 이해하는 과정"
- **meta/prompt-builder.md**: "좋은 프롬프트"에는 항상 "설명"이 포함

## 7가지 원칙의 상호연관성

```
원칙 1-2: 질문과 답변 + 데이터 기반
→ "어떤 질문에 데이터로 입증된 답변을?"

원칙 3-4: 추론 전략 + 워크플로우 체인
→ "AI의 생각 과정을 체계적으로, 사용자가 반복 학습하도록"

원칙 5-6: 자기 검증 + 레이어드 접근성
→ "신뢰도로 불확실성을 명시하고, 누구나 이해 가능하게"

원칙 7: 교육적 투명성
→ "모든 과정을 배움의 기회로 설계"
```

---

## 이 7가지 원칙을 적용한 AI PM의 업무 흐름

```
1. 기획 (Planning)
   원칙 1-2: "어떤 질문을 풀어야 하나?" (데이터로 입증)

2. 설계 (Design)
   원칙 3-4: "AI의 생각 과정 설계" (Reasoning + Workflow)

3. 출시 (Launch)
   원칙 5-6-7: "신뢰도 + 레이어드 + 교육적 공개"

4. 측정 (Measurement)
   원칙 2: "가설이 맞는가?" (데이터로 검증)

5. 개선 (Improvement)
   원칙 4: "워크플로우 피드백 → 모델 개선 반복"

6. 확대 (Scale)
   원칙 1: "새로운 질문/시나리오로 확대"

(다시 1→2→3→4→5→6 반복)
```

## 결론: AI PM의 철학

이 7가지 원칙을 관통하는 하나의 철학:

**"AI는 기술이 아니라 사용자를 돕는 '협력자'이고,
 AI PM은 이 협력이 자연스럽고 신뢰 가능하도록 설계하는 역할이다."**

즉:
- ❌ "AI 모델이 얼마나 강력한가?" (기술 중심)
- ✅ "사용자가 AI를 얼마나 믿고 활용하는가?" (인간 중심)

모든 프롬프트, 모든 의사결정, 모든 측정은 이 철학을 따른다.
